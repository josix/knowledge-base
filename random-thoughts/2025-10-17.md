## 如何從 Claude Code Session 生成技術部落格文章

### 工作流程概覽

整個流程包含五個主要步驟，採用 Claude Code 和 GPT-5 協作的方式，從開發 session 中整理出高品質的技術部落格文章。

### Context

一次 Claude Code session 中完成了特定目標或解決了某個問題，產出了可交付的工作成果。

### 步驟 1：知識整合與初稿撰寫

**Prompt**：
```
Consolidate a document that explains all the changes and related core concepts about xxxxx, along with references and resources, into a standalone Markdown blog article that records all the updates and how to apply them correctly, with a suitable title.
```

**輸出**：一篇包含完整技術內容的 markdown 文章

### 步驟 2：風格對齊與內容優化

**方式**：在個人 digital garden 或 blog 的 repo 目錄下開啟 Claude Code session

**Prompt 建議**：
```
請掃描這個 repo 中既有的文章，理解我的寫作風格。針對剛才產生的文章，請：
1. 增加更多 reasoning 和深入的解釋
2. 加入更多具體的範例說明
3. 調整結構和表達方式以符合我過往文章的風格
4. 確保技術深度和可讀性的平衡
```

這樣 Claude Code 可以：
- 自動讀取你過往的文章
- 理解你的寫作風格和習慣（語氣、段落結構、範例密度等）
- 根據相似主題的文章調整新內容的結構和表達方式
- 增加更多推理過程和詳細說明

**重點**：明確要求「more reasoning」和「具體舉例」能顯著提升文章品質。

### 步驟 3：多維度內容審查

**Prompt**：
```
Do fact check and review the concepts and content in detail
```

這個 prompt 會讓 AI 檢查：

1. **正確性 (Correctness)** - 技術描述、程式碼範例是否正確
2. **事實查核 (Fact Check)** - 引用的文件、版本資訊是否正確
3. **概念準確性** - 核心概念的解釋是否精確
4. **內容完整性** - 是否有遺漏的重要資訊

**工具使用建議**：
- **Claude Code**：主要負責撰寫和修改內容
  - 開啟 thinking mode 以獲得更深入的推理
  - 使用 web search 驗證最新文件和版本資訊
- **GPT-5 (ChatGPT Plus)**：負責確認正確性並提供回饋
  - 從不同角度審查內容
  - 提供改進建議和補充觀點

**協作模式**：
1. 用 GPT-5 審查文章，產生具體的改進建議和問題清單
2. 將建議帶回 Claude Code session 進行修改
3. Claude Code 執行修改並解決問題

### 步驟 4：迭代式改進

讓 GPT-5 和 Claude Code 進行多輪迭代：

1. **GPT-5 審查回合**：提供改進建議、發現不一致或錯誤
2. **Claude Code 修改回合**：根據建議執行修改
3. **人工判斷**：適當地去掉一些修改或回饋當你覺得不需要的時候

**注意事項**：
- 保持批判性思考，不要盲目接受所有 AI 建議
- 技術正確性永遠優先於文字優美
- 確保保留你個人的聲音和風格
- 通常 2-3 輪迭代即可達到良好品質

**迭代範例**：
- 第一輪：修正技術錯誤、補充遺漏的概念
- 第二輪：改善範例品質、增加實際應用情境
- 第三輪：潤飾文字、優化結構流暢度

### 步驟 5：細節完善與發布準備

所有的小建議和小問題都解決後，進行最後的細節檢查：

**檢查清單**：
- [ ] 修正標點符號和排版問題
- [ ] 檢查程式碼範例的完整性和可執行性
- [ ] 確認所有超連結正常運作
- [ ] 驗證 front matter 資訊正確（date, description, tags）
- [ ] 確認程式碼區塊的語法高亮標記正確
- [ ] 檢查中英文混排的空格處理
- [ ] 驗證技術術語的一致性
- [ ] 評估文章可讀性指標（見下方說明）

#### 可讀性評估：Flesch Reading Ease

**什麼是 Flesch Reading Ease？**

Flesch Reading Ease 是一個量化文章可讀性的指標，分數範圍 0-100：

- **90-100**：非常容易（小學五年級程度）
- **60-70**：標準（國高中程度）
- **30-50**：困難（大學程度）
- **0-30**：非常困難（研究生程度）

**計算公式**：

```text
206.835 - 1.015 × (總字數/總句數) - 84.6 × (總音節數/總字數)
```

**技術文章的理想範圍**：

- **教學型文章**：建議 50-60（易讀但保有深度）
- **進階技術文章**：建議 30-50（允許較複雜的表達）
- **API 文件**：建議 60-70（力求簡潔明確）

**如何使用 AI 協助評估**：

```text
請評估這篇文章的 Flesch Reading Ease 分數，並提供改進建議：
1. 計算目前的可讀性分數
2. 指出過於複雜的句子（超過 25 字）
3. 建議如何簡化而不失技術準確性
4. 保持技術深度的前提下提升可讀性
```

**實用技巧**：

- 用短句取代長句（一句話一個重點）
- 避免過多子句嵌套
- 適當使用列表和分段
- 在複雜概念後加入具體範例
- 使用主動語態而非被動語態

**注意事項**：

- 技術正確性永遠優先於可讀性分數
- 某些技術術語無法簡化，這是正常的
- 不要為了提高分數而犧牲必要的細節
- 中文內容可能需要調整評估標準（原公式為英文設計）

### 關鍵成功因素

1. **充分利用不同 AI 的優勢**
   - **Claude Code (VSCode Extension)**：
     - 處理程式碼相關內容
     - 執行結構性修改
     - 整合 repo 上下文
     - 使用 thinking mode 進行深度推理
     - 使用 web search 驗證最新資訊
   - **GPT-5 (ChatGPT Plus)**：
     - 文字潤飾和流暢度改善
     - 從不同視角審查內容
     - 發現邏輯漏洞和不一致
     - 提供讀者視角的回饋

2. **保持批判性思考**
   - 不要盲目接受 AI 的所有建議
   - 技術正確性永遠優先於文字優美
   - 保持個人聲音和風格
   - 人工判斷哪些建議真正有價值

3. **時間成本**：約 65-95 分鐘可完成一篇高品質技術文章（相較傳統方式節省 60-70% 時間）

4. **品質標準**
   - 技術內容經過 fact check 驗證
   - 程式碼範例實際可執行
   - 包含充分的 reasoning 和解釋
   - 風格與既有文章一致

### 工具鏈配置

**必要工具**：
- **Claude Code**：VSCode Extension 版本
- **GPT-5**：透過 ChatGPT Plus 使用

**建議設定**：
- Claude Code 開啟 thinking mode（深度推理）
- Claude Code 啟用 web search（驗證最新文件）
- 在 blog/digital garden repo 目錄下工作（獲得上下文）

這個工作流程也可以應用於：

- 學習筆記整理
- 問題解決記錄（debugging、troubleshooting）
- 最佳實踐文件
- 技術分享簡報
- API 文件撰寫
- 開源專案的 README 和教學文件
